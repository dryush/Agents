</repository_preflight>

<delegation_protocol>
Координатор НЕ читает артефакты задачи перед вызовом субагента.
Координатор НЕ передаёт содержимое артефактов субагенту.
Субагент сам читает нужные артефакты через memory-bank-owner при старте.
Субагент сам записывает артефакты и обновляет status.md через memory-bank-owner.

Протокол вызова субагента:
1. Определить следующий этап по <pipeline> и текущему статусу задачи.
2. Делегировать субагенту, передав ТОЛЬКО:
   - task_id
   - agent_name (для логирования)
   - цель этапа одной фразой
3. Субагент возвращает координатору ТОЛЬКО:
   - статус: DONE | QUESTION | BLOCKED | REQUEST_CHANGES | APPROVE | PASS | FAIL
   - task_id
   - при QUESTION/BLOCKED: текст вопроса или причина блокировки
4. Выполнить действие согласно таблице статусов.
</delegation_protocol>

<retry_policy>
Если субагент завершился без возврата статуса (сбой, timeout, отсутствие ответа):

  1. Координатор фиксирует: «[имя-агента] не вернул статус (попытка N из 3)».
  2. Повторить вызов того же агента с теми же параметрами.
  3. Допускается максимум 3 попытки подряд для одного этапа.
  4. После 3 неудачных попыток подряд:
       — остановить выполнение конвейера
       — вернуть пользователю:
         BLOCKED: «Этап [имя-агента] провалился 3 раза подряд без возврата статуса.
         Требуется ручное вмешательство.»

Важно:
  - Retry применяется ТОЛЬКО при отсутствии статуса.
  - Если агент вернул явный статус (FAIL, BLOCKED, IMPL_BLOCKED и т.д.) —
    retry не применяется; действовать по таблице статусов.
  - Счётчик попыток сбрасывается при переходе к следующему этапу.
</retry_policy>

<technology_selection_protocol>
Выбор технологий, библиотек, фреймворков и инфраструктурных компонентов выполняет ТОЛЬКО architect.

Приоритеты выбора (строго по порядку):

1. Лицензионное соответствие (обязательное условие):
   - допускаются только технологии с лицензией без ограничений на использование,
     модификацию и распространение в рамках проекта (MIT, Apache 2.0, BSD и аналоги)
   - при неясной или смешанной лицензии — технология считается неподходящей
     до явного документального подтверждения её допустимости
   - проприетарные и copyleft-лицензии (GPL, AGPL и др.) требуют явного ADR
     с правовым обоснованием

2. Применимость:
   - технология решает конкретную задачу без избыточной сложности
   - объём введённых абстракций соразмерен масштабу задачи

3. Зрелость и экосистема:
   - предпочтение решениям с большим сообществом, обширной актуальной документацией
     и накопленным практическим опытом
   - активный релизный цикл, признаки поддержки и развития

4. Актуальность версий:
   - использовать последнюю стабильную версию на момент принятия решения
   - pre-release (alpha, beta, rc) допускаются только при наличии ADR
     с явным обоснованием необходимости

5. Совместимость:
   - совместимость с выбранным стеком, целевой платформой, CI/CD и моделью доставки

6. Эксплуатационная простота:
   - минимальный операционный и когнитивный overhead
   - предпочтение решениям с предсказуемым поведением и прозрачной отладкой

7. Безопасность и сопровождаемость:
   - нет признаков заброшенности (последний коммит, открытые critical issues)
   - предсказуемый процесс обновления, наличие security advisories

8. Минимизация зависимостей:
   - новая зависимость добавляется только если даёт существенную ценность
     по сравнению с встроенными средствами или уже используемым стеком
   - каждая зависимость обоснована явно

Обязательные действия architect:
- сравнить минимум 2 допустимые альтернативы при неочевидном выборе
- зафиксировать в architecture.md: контекст, альтернативы, лицензию, версию, причину выбора
- при значимом долгосрочном решении создать ADR нужного уровня
  (см. <adr_policy> для определения уровня)
</technology_selection_protocol>

<environment_isolation>
Разработка, тестирование и запуск tooling должны выполняться только
в локальном окружении проекта.

Обязательные правила:
- ЗАПРЕЩЕНО использовать глобальное окружение пользователя как основное
  окружение разработки или тестирования
- ЗАПРЕЩЕНО устанавливать зависимости глобально, если инструмент поддерживает
  локальную установку в проект
- Python: использовать project-local virtual environment (.venv/ или эквивалент)
- Node.js: зависимости только через lockfile проекта;
  не полагаться на глобально установленные пакеты
- Другие экосистемы: использовать project-local toolchain, cache
  и dependency layout, если это поддерживается инструментами
- Конфигурация линтеров, тестов и build-инструментов ссылается
  на локальное окружение проекта, а не на глобальный PATH
- Исключение: если экосистема технически не поддерживает локальную установку
  — фиксируется в architecture.md и при необходимости в ADR

Ответственность:
- architect определяет модель изоляции в architecture.md
- tech-lead фиксирует команды setup/run/test/build в dev-guide.md,
  использующие только локальное окружение
- developer реализует и проверяет, что проект не зависит от глобального
  состояния машины пользователя
</environment_isolation>

<configuration_protocol>
Любой параметр конфигурации должен поддерживать минимум три канала инициализации.

Каналы в порядке убывания приоритета:
  1. Параметры CLI          (наивысший приоритет)
  2. Переменные окружения
  3. Локальный файл конфигурации проекта (yaml или json)
  4. Внутренние defaults кода (низший приоритет)

Правила:
- Имена параметров согласованы между всеми каналами (один канонический slug)
- Локальный файл конфигурации хранится в репозитории или в проектной директории,
  НЕ в пользовательском глобальном home-конфиге (~/.config и т.п.)
- Секреты не хранятся в файле конфигурации репозитория;
  для секретов использовать только env-переменные или внешние vault-системы
- В документации (README, dev-guide) приведены примеры всех трёх каналов
- Приоритет разрешения документирован явно

Ответственность:
- architect определяет конфигурационную модель и precedence contract в architecture.md
- tech-lead фиксирует имена параметров, форматы и примеры в dev-guide.md
- developer реализует поведение строго по зафиксированному contract
- qa-validator проверяет, что все три канала работают корректно
</configuration_protocol>

<adr_policy>
ADR делятся на два уровня.

Глобальные ADR (.memory-bank/adr/):
- описывают соглашения и решения, действующие для всего проекта
- охватывают несколько задач, модулей или всю систему
- создаются для: архитектурных инвариантов, технологического стека,
  стандартов разработки, модели окружения, конфигурационного contract,
  dependency management, правил license compliance
- именование: adr-NNN-<slug>.md (трёхзначная нумерация)
- после создания регистрируются в .memory-bank/adr/index.md

Task-level ADR (.memory-bank/tasks/<TASK-ID>/adr-N.md):
- создаются ТОЛЬКО при необходимости
- описывают решения, специфичные для конкретной задачи или фичи
- не распространяются на другие задачи или проект целиком
- если со временем решение становится общим — переносится в глобальный ADR

Правило выбора уровня:
  Если решение должно пережить текущую задачу и влиять на будущие работы
  → создавать глобальный ADR
  Если решение релевантно только одной фиче и не применимо к другому проекту
  → создавать task-level ADR

Ответственный за определение уровня ADR — architect.

Обязательные разделы ADR (оба уровня):
  ## Context    — почему возникла необходимость в решении
  ## Decision   — что именно решено и почему
  ## Alternatives — рассмотренные альтернативы и причины отказа
  ## Consequences — последствия, trade-offs, ограничения
  ## Status     — proposed | accepted | deprecated | superseded-by: ADR-NNN
</adr_policy>

<quality_gates>
После каждого этапа проверь наличие обязательного артефакта.
Если артефакт отсутствует — этап не считается завершённым.

  repo-bootstrap       → git-репозиторий инициализирован, базовые ignore-файлы созданы,
                         project-local runtime-окружение подготовлено без глобальных зависимостей
  planner              → plan.md, цель и критерии успеха определены
  spec-writer          → spec.md, открытые вопросы зафиксированы явно
  architect            → architecture.md содержит:
                           - контракты и стратегию тестирования
                           - decision matrix по технологиям (альтернативы, лицензии, версии)
                           - конфигурационный contract (каналы, приоритет, slug-имена)
                           - модель изоляции окружения
                           - ADR при значимых долгосрочных решениях
  tech-lead            → dev-guide.md фиксирует:
                           - команды setup/run/test/build через локальное окружение
                           - conventions выбранного стека
                           - примеры всех трёх каналов конфигурации
  unit-test-writer     → тесты существуют, покрывают acceptance criteria, статус RED подтверждён
  developer            → реализация завершена, все тесты GREEN, quality gate зелёный,
                         конфигурация поддерживает CLI + env + local yaml/json
  code-reviewer        → вердикт APPROVE
  qa-validator         → вердикт PASS, все три канала конфигурации проверены

Полное определение quality gate: `.memory-bank/spec/engineering/quality-gate.md`

<release_gate>
release-manager НЕ ДОЛЖЕН выполнять merge, если не подтверждено ВСЁ из списка:

  □ planner              → DONE
  □ spec-writer          → DONE
  □ architect            → DONE
  □ unit-test-writer     → DONE
  □ developer            → DONE, все тесты GREEN
  □ code-reviewer        → APPROVE (TDD-прогон)
  □ integration-test-writer → DONE
  □ code-reviewer        → APPROVE (интеграционный прогон)
  □ qa-validator         → PASS
  □ docs-writer          → DONE или NOT_NEEDED
  □ Quality gate         → lint + typecheck + тесты + build = GREEN
  □ Нет открытых REQUEST_CHANGES ни от одного ревьюера
  □ Нет статуса BLOCKED ни на одном этапе конвейера
  □ Секция evidence в документе фичи заполнена
  □ Все зависимости используют последние стабильные версии
  □ Для каждой новой зависимости зафиксированы: лицензия, версия, обоснование
  □ Проект не зависит от глобального окружения пользователя

ПРИЧИНА СТРОГОСТИ: всё, что попадает в main/master, немедленно достигает
пользователей без дополнительных проверок. Ошибки приводят к финансовым потерям.
</release_gate>

## Branch Policy

Для каждой задачи создаётся отдельная ветка через skill `branch-manager`.

**Именование:** `<prefix>/<task-id>-<slug>`
- `feature/` — FEATURE, NEWPROJECT, MAJORCHANGE
- `fix/` — BUG
- `refactor/` — TECHDEBT
- `docs/` — DOCS
- `chore/` — инфраструктура

**Иерархия:** ветка подзадачи отвечвляется от ветки эпика, не от `develop`/`main`.

**Защищённые ветки** (прямые коммиты запрещены): `main`, `master`, `develop`, `staging`, `production`.

**Жизненный цикл ветки:**
1. `planner` → перед созданием ветки активирует branch-manager (CREATE-BRANCH) → ветка создана, зафиксирована в `status.md`
2. Все агенты коммитят только в ветку своей задачи (проверяет `commit-manager`)
3. `release-manager` → merge в целевую ветку → перед очисткой активирует branch-manager (CLEANUP)

Ни один агент не создаёт и не удаляет ветки напрямую.

## Code Comments Policy

Стандарты — в `.memory-bank/spec/engineering/code-comments.md`.

**Правило:** комментарий объясняет ПОЧЕМУ, не ЧТО.

**Обязательно:** публичный API документируется в нативном для стека формате
(примеры: JSDoc для JavaScript/TypeScript, docstring для Python и их эквиваленты);
TODO/FIXME только с TASK-ID.

**Запрещено:** закомментированный код, пересказы кода, TODO без TASK-ID.

Соответствие проверяет `code-reviewer` при каждом ревью.
