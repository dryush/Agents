# Quality Gate

<rule>
Quality gate выполняется в строго фиксированном порядке.
Остановка при первом провале — следующие шаги не запускаются.
Конкретные команды для каждого шага — в dev-guide.md проекта.
</rule>

<steps>
ШАГ 1 — lint
  Проверяет: стилевые и синтаксические нарушения.
  Критерий: ноль ошибок (warnings допустимы только если явно разрешены в dev-guide.md).
  Провал → зафиксировать файлы и строки, остановить gate.

ШАГ 2 — typecheck
  Проверяет: типовую корректность (TypeScript, mypy, etc.).
  Критерий: ноль ошибок типизации.
  Провал → зафиксировать ошибки, остановить gate.

ШАГ 3 — unit tests
  Проверяет: изолированную корректность каждого модуля.
  Критерий: 100% тестов PASS, ноль пропущенных (skipped).
  Провал → зафиксировать упавшие тесты, остановить gate.

ШАГ 4 — build
  Проверяет: сборка артефакта без ошибок.
  Критерий: build завершился с exit code 0.
  Провал → зафиксировать вывод build, остановить gate.

ШАГ 5 — integration tests  [если существуют]
  Проверяет: взаимодействие модулей и адаптеров.
  Критерий: 100% тестов PASS, ноль пропущенных.
  Провал → зафиксировать упавшие тесты.
</steps>

<green_definition>
Quality gate считается ЗЕЛЁНЫМ только при:
  - все шаги завершены без ошибок
  - ноль пропущенных тестов
  - ноль подавленных ошибок lint или typecheck

Частичный зелёный (например, только lint и typecheck) — НЕ зелёный.
</green_definition>

<reporting>
При провале агент фиксирует:
  - какой шаг упал
  - вывод команды (полный или первые 50 строк)
  - список затронутых файлов
  - предполагаемая причина (если очевидна)

Результат включается в возвращаемый координатору артефакт или статус.
</reporting>
