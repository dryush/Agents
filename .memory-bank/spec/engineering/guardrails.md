<architecture>
## Feature-Slice Design (FSD) с Application Layer

Структура проекта организована по слоям и срезам.
Application layer присутствует ВСЕГДА — это основной слой бизнес-логики.

  src/
  ├── shared/           ← утилиты, UI-kit, константы, типы без бизнес-логики
  │   └── lib/logger/   ← корневой логгер (единственный источник logger)
  ├── entities/         ← бизнес-сущности (User, Campaign, Survey...)
  ├── application/      ← ВСЕ действия системы: use-cases, команды, запросы
  │   └── <action>/
  │       ├── index.ts  ← публичный API
  │       ├── action.ts ← реализация use-case
  │       └── action.test.ts
  ├── features/         ← адаптеры application для конкретных UI-сценариев
  ├── widgets/          ← композиция features для UI-блоков
  ├── pages/            ← композиция widgets для экранов
  └── processes/        ← межстраничные процессы

Каждый slice содержит только то, что нужно ему самому:

  features/submit-feedback/
  ├── index.ts     ← публичный API (единственная точка импорта)
  ├── ui/          ← компоненты
  ├── model/       ← стейт, сторы, хуки (тонкая оболочка над application/)
  ├── api/         ← адаптеры к внешним сервисам
  └── lib/         ← вспомогательные утилиты slice
</architecture>

<application_layer>
## Application Layer — обязательные правила

Application layer описывает ВСЕ действия, которые может совершать система.

КРИТЕРИЙ ГОТОВНОСТИ application layer:
  Для каждого use-case должен существовать или быть реализуем
  неинтерактивный CLI-интерфейс (команда → ответ) без изменения application/.
  Если use-case нельзя вызвать из CLI без рефакторинга — application layer не готов.

ПРАВИЛА:
  - Каждый use-case — отдельный файл, одна публичная функция.
  - Use-case принимает явные параметры (не HTTP-запрос, не DOM-событие).
  - Use-case возвращает явный типизированный результат или ошибку.
  - Use-case не знает о транспорте: нет req/res, нет React-хуков, нет DOM.
  - Транспортные адаптеры (HTTP, CLI, WebSocket) вызывают application/, не наоборот.
  - Состояние хранится в application/ или entities/ — не в UI.

ЗАПРЕЩЕНО в application/:
  - импорт из features/, widgets/, pages/, processes/
  - обращение к DOM, window, document
  - прямые вызовы HTTP/fetch (только через порт-интерфейс)
</application_layer>

<dependency_rule>
## Правило зависимостей (строгое)

Допустимые направления импорта:

  pages → widgets → features → application → entities → shared

Запреты:
  - импорт из вышележащего слоя — ЗАПРЕЩЁН
  - кросс-импорт внутри слоя — ЗАПРЕЩЁН (feature/a не импортирует из feature/b)
  - импорт минуя index.ts чужого slice — ЗАПРЕЩЁН
    плохо:  import { helper } from "features/auth/lib/utils"
    хорошо: import { helper } from "features/auth"

Нарушение требует ADR.
</dependency_rule>

<logging_tracing>
## Логирование, трассировка и мониторинг

Логирование строится на ОДНОМ корневом логгере из shared/lib/logger/.
Все остальные модули получают child-логгер через dependency injection.

УРОВНИ (строго в этом порядке):
  TRACE  — детальный поток выполнения (только для отладки)
  DEBUG  — диагностическая информация
  INFO   — нормальные события системы (запуск use-case, результат)
  WARN   — аномалия без сбоя (устаревший параметр, медленный запрос)
  ERROR  — сбой операции, система продолжает работу
  FATAL  — критический сбой, система не может продолжать

СТРУКТУРНЫЙ ЛОГ — обязателен. Каждая запись лога — JSON-объект с полями:
  {
    "level":      "INFO",
    "ts":         "ISO-8601",
    "traceId":    "uuid-v4",        ← сквозной ID запроса / операции
    "spanId":     "uuid-v4",        ← ID текущего шага
    "module":     "application/submitFeedback",
    "action":     "submit",
    "durationMs": 42,               ← обязателен на INFO и выше
    "msg":        "feedback submitted",
    "data":       {}                ← опционально, без PII
  }

СКВОЗНОЙ ИДЕНТИФИКАТОР (traceId):
  - Генерируется на входе системы (HTTP-обработчик, CLI-команда, cron).
  - Передаётся через все слои явным параметром или context-объектом.
  - Присутствует в КАЖДОЙ записи лога данной операции.
  - Не хранится в глобальном состоянии.

КАТЕГОРИЗАЦИЯ по модулям:
  Поле "module" содержит путь slice: "application/submitFeedback",
  "entities/campaign", "shared/lib/mailer". Позволяет фильтровать логи
  по слою и конкретному use-case.

ПРАВИЛА:
  - Каждый use-case логирует старт (DEBUG) и завершение (INFO) с durationMs.
  - Каждая ошибка логируется на уровне ERROR с полным stack trace.
  - PII (email, имя, пароль) — ЗАПРЕЩЕНЫ в логах на любом уровне.
  - Внешние зависимости (БД, HTTP) логируются адаптером, не use-case.
  - В тестах логгер мокируется; сайд-эффекты логирования не проверяются
    если не тестируется сам логгер.

МОНИТОРИНГ:
  - Счётчики (вызовы use-case, ошибки) экспортируются через порт-интерфейс
    MetricsPort в shared/lib/metrics/.
  - Реализация MetricsPort — в infrastructure (Prometheus, StatsD, etc.).
  - В тестах MetricsPort замокирован.
</logging_tracing>

<ui_rules>
## UI — тонкий адаптер

UI не хранит бизнес-состояние и не содержит бизнес-логику.

ПРАВИЛА:
  - UI-компонент вызывает use-case из application/ и отображает результат.
  - Локальное UI-состояние допустимо только для: загрузка, ошибка отображения,
    анимация, видимость элемента.
  - Бизнес-состояние (данные кампании, список участников, результаты опроса)
    хранится в application/ или entities/, не в useState/useStore UI-компонента.
  - UI не трансформирует данные — только форматирует для отображения.
  - Проверяемость: каждый use-case должен быть тестируем без рендера UI.
</ui_rules>

<kiss>
## KISS — Keep It Simple, Stupid

ЯСНОСТЬ:
  - явный код предпочтительнее краткого
  - нет «умных» абстракций, которые нельзя понять с первого прочтения
  - нет преждевременной обобщённости (YAGNI)
  - дублирование лучше неправильной абстракции

РАЗМЕР: файл и функция должны читаться целиком без прокрутки контекста.
  Сигнал к разбиению — необходимость держать в голове содержимое
  нескольких экранов одновременно для понимания одной функции.

СЛОЖНОСТЬ: минимальная глубина вложенности. При появлении глубоко
  вложенных условий — рефакторинг через early return или выделение функции.

ИМЕНОВАНИЕ:
  - имя функции описывает действие: getUser, submitFeedback, formatDate
  - имя переменной описывает содержимое: userList, isLoading, campaignId
  - нет сокращений кроме общепринятых (id, url, api, dto)
</kiss>

<slice_contract>
## Контракт публичного API slice

Каждый slice экспортирует ТОЛЬКО через index.ts то, что нужно другим слоям.
Внутренние файлы slice НЕ импортируются снаружи напрямую.

Пример нарушения:
  import { validateEmail } from "features/auth/lib/validators"  // ЗАПРЕЩЕНО
Исправление:
  import { validateEmail } from "features/auth"  // через index.ts
</slice_contract>

<type_safety>
## Типизация

  - Строгая типизация обязательна во всех слоях.
  - any, object, unknown без явного приведения — ЗАПРЕЩЕНО.
  - Типы данных API (DTO) определяются в slice api/ и не утекают в model/ или ui/.
  - Преобразование DTO → доменный тип — на границе api/.
  - Переменные окружения — только в shared/config/ или slice api/.
</type_safety>

<external_dependencies>
## Внешние зависимости

  - Вызовы API, БД, файловой системы — только в slice api/.
  - Бизнес-логика в application/ не знает о механизме доставки данных.
  - В тестах api/ всегда замокирован.
  - Прямые вызовы внешних сервисов из ui/ или application/ — ЗАПРЕЩЕНЫ.
</external_dependencies>

<testing_constraints>
## Тестирование

  - unit-тесты: изолируют один use-case или один файл внутри slice.
  - Тест не зависит от реального окружения (БД, сеть, FS).
  - Тест детерминирован: одни входные данные → всегда один результат.
  - Тест проверяет поведение через публичный интерфейс (index.ts).
  - Developer НЕ изменяет тесты — только реализацию.
</testing_constraints>

<context_budget>
## Бюджет контекста LLM

  - Один PR / одна задача затрагивает не более нескольких slices.
  - Один slice изменяется атомарно: ui + model + api вместе, не по отдельности.
  - Новая сущность или feature — новый slice, не расширение существующего.
  - Если изменение требует одновременного понимания большого числа файлов —
    сигнал разбить задачу на подзадачи.
</context_budget>

<adr_triggers>
## Обязательные ADR

Создать ADR при:
  - нарушении правила зависимостей между слоями
  - введении кросс-импорта внутри слоя
  - бизнес-логике в UI-слое
  - хранении бизнес-состояния в UI
  - use-case, не вызываемом из CLI без рефакторинга
  - отключении правил типизации или lint
  - отказе от структурного логирования
  - любом отступлении от правил этого файла

ADR размещается в: .memory-bank/tasks/<TASK-ID>/adr-<номер>.md
</adr_triggers>
