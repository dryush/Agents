# Code Comments Standards

<principle>
Комментарий объясняет ПОЧЕМУ, не ЧТО.
Код объясняет что делается — комментарий объясняет намерение, контекст или ограничение.
Комментарий, пересказывающий код, — шум. Его не должно быть.
</principle>

<when_to_comment>
ОБЯЗАТЕЛЬНО комментировать:
  - нетривиальное бизнес-правило ("почему именно 7 дней, а не 30")
  - обходное решение с TASK-ID: // TODO [BG-0042]: убрать после фикса upstream
  - публичный API модуля (см. <api_docs>)
  - неочевидное граничное условие или допущение

ЗАПРЕЩЕНО комментировать:
  - очевидный код: i++ // increment i
  - закомментированный код без объяснения — удалять, не комментировать
  - шаблонные фразы: "// Constructor", "// Getters"
  - TODO без TASK-ID: // TODO: fix this later — ЗАПРЕЩЕНО
</when_to_comment>

<api_docs>
Публичные функции, классы и методы требуют doc-комментария.
Формат по языку:

TypeScript / JavaScript — JSDoc:
```
/**
 * Краткое описание в одну строку.
 *
 * @param paramName - описание параметра
 * @returns описание возвращаемого значения
 * @throws {ErrorType} когда выбрасывается ошибка
 *
 * Ограничения и неявные зависимости, если есть 
 */
```

Python — Google-style docstring:
```
def func(param: Type) -> ReturnType:
    """Краткое описание в одну строку.

    Args:
        param: описание параметра.

    Returns:
        описание возвращаемого значения.

    Raises:
        ErrorType: когда выбрасывается ошибка.
        
    Ограничения и неявные зависимости, если есть 
    """
```

Исключения (doc-комментарий не нужен):
  - приватные вспомогательные функции с очевидным назначением
  - тестовые функции — их именование самодокументирует
</api_docs>

<inline_comments>
Inline-комментарий (в строке кода):
  - Только ПОЧЕМУ, никогда ЧТО.
  - Два пробела от кода.
  - Максимум одна строка.
  - При workaround ссылка на задачу обязательна: // [BG-0042] upstream bug

Блочный комментарий (над кодом):
  - Для нетривиального алгоритма или группы строк.
  - Максимум 3-5 строк.
  - Отделён пустой строкой сверху и снизу.
</inline_comments>

<todo_format>
Все TODO/FIXME/HACK содержат TASK-ID:

  // TODO [FT-0042]: реализовать кеширование после завершения задачи
  // FIXME [BG-0017]: временный обход до фикса в upstream
  // HACK [TD-0003]: убрать при рефакторинге слоя infrastructure

TODO без TASK-ID — нарушение, выявляемое code-reviewer.
</todo_format>

<enforcement>
Проверяет code-reviewer:
  [] нет закомментированного кода
  [] нет TODO без TASK-ID
  [] публичный API задокументирован (JSDoc / docstring)
  [] нет комментариев-пересказов кода
  [] inline-комментарии объясняют ПОЧЕМУ
</enforcement>
