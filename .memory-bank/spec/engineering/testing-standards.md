# Testing Standards

<tdd_cycle>
Обязательная последовательность для каждой функциональной единицы:

  RED    → тест написан, падает по ожидаемой причине (не compile error)
  GREEN  → минимальная реализация, тест проходит
  REFACTOR → код улучшен, тесты остаются GREEN

Пропуск фазы RED (тест написан после реализации) — нарушение стандарта.
</tdd_cycle>

<coverage_matrix>
Каждый acceptance criterion покрывается четырьмя типами тестов:

  happy path   — корректный ввод, ожидаемый результат
  alt path     — корректный ввод, альтернативный результат (ветки логики)
  edge case    — граничные значения, пустые коллекции, нули, максимумы
  error case   — некорректный ввод, недоступные зависимости, таймауты

Отсутствие любого типа для критерия — неполное покрытие.
</coverage_matrix>

<test_structure>
Именование тестового файла:
  unit:         <module>.test.<ext>  или  <module>.spec.<ext>
  integration:  <feature>.integration.test.<ext>
  e2e:          <scenario>.e2e.test.<ext>

Именование тест-кейса:
  <unit_under_test> <action> <expected_result>
  Пример: "createUser returns error when email already exists"

Структура тест-кейса (AAA):
  Arrange  — подготовить входные данные и моки
  Act      — вызвать тестируемый код
  Assert   — проверить результат и побочные эффекты

Один тест-кейс проверяет одно утверждение.
</test_structure>

<test_levels>
unit:
  - изолирует один модуль или функцию
  - все внешние зависимости замокированы
  - выполняется без сети, БД, файловой системы
  - скорость: < 100ms на тест

integration:
  - проверяет взаимодействие 2+ модулей
  - внешние зависимости замокированы на уровне адаптера
  - выполняется без реальных внешних сервисов

e2e:
  - проверяет сценарий целиком от входа до выхода
  - допускает реальные зависимости только в изолированном тестовом окружении
  - запускается отдельно от unit и integration
</test_levels>

<forbidden>
- Тест с жёстко заданным временем (sleep, fixed timestamp) без мока времени.
- Тест, зависящий от порядка выполнения других тестов.
- Тест, изменяющий глобальное состояние без очистки в teardown.
- Пропуск теста (skip/xtest/xit) без комментария с TASK-ID и причиной.
</forbidden>

<modular_tdd>
Модульная система TDD описывает порядок декомпозиции и исполнения под-задач
в рамках TDD-цикла. Полный процесс — в tdd-modular-process.md.

Ключевые правила:
1. До начала TDD-цикла unit-test-writer строит dependency-map.md и tdd-task-list.md.
   Входные данные: plan.md со статусом FINAL (список под-задач от architect).
2. Под-задача — один класс или одна логически связанная группа методов.
3. Под-задача блокирована, если хотя бы одна из её зависимостей не имеет статуса DONE.
4. developer запускает тесты строго в соответствии с test-scope текущей под-задачи.
5. После завершения каждой под-задачи — smoke-запуск тестов затронутых feature.
6. После завершения всех под-задач эпика — запуск тестов затронутых epic.
7. Полный набор тестов — только перед merge в main (через release-manager).

Инструменты (скрипты) реализует tech-lead в dev-guide.md с учётом конкретного стека:
  - скрипт определения test-scope для модуля (по маппингу из dependency-map.md)
  - скрипт запуска тестов по scope (module / feature / epic / full)
  - скрипт валидации dependency-map.md (нет циклов, все зависимости разрешимы)
</modular_tdd>
